# 晴川知识库 · 人机协作技能

> 一个帮你「长」出认知体系的 Claude Code / OpenClaw Skill。
> 不是整理笔记的工具——是基于验证次数的认知生长系统。

## 这不是什么

这不是另一个「把想法放进正确文件夹」的知识管理工具。那是图书馆思维——知识是静态的物件，管理就是找到它的位置。

一个想法今年记下来，明年回头看你会发现它变了。不是你记错了，是你长大了。

## 这是什么

一个基于 **验证次数** 的四层认知生长模型：

```
碎片(日记) → 器(方法论) → 不器(洞察) → 所以为人(世界观)
```

传统工具问「这是什么？」→ 放到对应文件夹。
这里问的是**「你验证过几次？」**

- 只经历过一次 → 日记
- 做过三次以上 → 提炼成方法论（器/）
- 多个方法论浮现模式 → 写成洞察（不器/）
- 洞察改变了「为什么做这件事」→ 更新世界观（所以为人/）

每一层不是标签，是你对这件事的**确信程度**。升级不靠感觉，靠验证次数。

## 核心理念：君子不器

三层结构的命名来自「君子不器」：

- **器/** — 工具层。学会方法、掌握技能。`verified_count` 门槛确保你不会满足于当一个工具人——验证够了就推你向上走。
- **不器/** — 打破工具边界。从多个方法论里看到模式。「游戏行为分析的方法竟然也能解释学习拖延」——这就是不器的时刻。`contradicts` 矛盾字段强制存在：洞察不是因为没看到反面才成立，是因为把反面也装进去了才坚固。
- **所以为人/** — 最终指向。洞察积累到改变了「你为什么做这些、你是什么样的人」。

## 三个反直觉的选择

**① 按验证次数分层，不按主题分类。** 「别人能照着做吗？」→ 器/。「能解释 ≥3 件已发生的事吗？」→ 不器/。「5 年后还重要吗？」→ 所以为人/。工程现实主义：不靠感觉说懂了，靠验证次数。

**② MOC 是阅读路径，不是目录。** 200+ 篇笔记的文件清单毫无用处。MOC 告诉你「想解决 X 问题 → 读 A→B→C」，且写到一半缺了某篇笔记时，MOC 本身就暴露了知识缺口。

**③ 校准和生长是两条独立回路。** 校准问「我偏了吗？」（行动偏离方向 → 拉回来）。生长问「我变了吗？」（新洞察挑战旧认知 → 方向锚本身进化）。大多数系统只做第一条，但雏形世界观需要在学习中长出来，不能被一直锁在早期锚上反复纠偏。

→ 完整设计理念见 [reference/design-rationale.md](reference/design-rationale.md)

## 快速安装

```bash
# 一键安装到 Claude Code
curl -fsSL https://raw.githubusercontent.com/chenyitao123/knowledge-base-habits/main/install.sh | bash
```

或手动：

```bash
git clone https://github.com/chenyitao123/knowledge-base-habits.git
cd knowledge-base-habits
bash install.sh
```

安装时会交互式询问 Obsidian 路径、AI 称呼等偏好。

## 使用方式

| 你说 | 晴川做什么 |
|------|-----------|
| 「记一下」/「帮我记录一下」 | 日记双写 + Tasks 追加 |
| 「这个放哪」 | 判断落位 + 写 frontmatter + 建立连接 |
| 「帮我看看方向」 | 对比方向锚定 vs 行动，诊断偏离 |
| 「帮我回顾」/周日 | 检查升级条件，提炼碎片为方法论/洞察 |
| 「专栏写一篇」 | 世界观层输出长文 |

方向决策、关键资源更新、重大认知转折时**自动触发**记录，无需手动说「记一下」。

## 底层信念

知识管理的本质不是分类，是用经验验证认知。你真正知道的不是你能背出来的东西，是你做过三次以上、看到过模式、它改变了你决策方式的东西。
