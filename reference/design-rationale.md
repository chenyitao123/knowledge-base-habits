# 底层逻辑与认知

> 需要时加载：`read skills/knowledge-base-habits/reference/design-rationale.md`

## 为什么是「生长路径」而不是「分类系统」？

传统的知识管理 = 把东西放进正确的文件夹。但知识不是静态的——它会生长、会迭代、会推翻重来。

碎片 → 器 → 不器 → 所以为人 这条路径对应的是**认知的成熟度**：
- 你经历过一次 → 日记
- 你做过三次以上 → 方法论
- 你从多个方法论中看到了模式 → 洞察
- 洞察积累到改变了你对「自己为什么做这件事」的理解 → 世界观

每一层都不是分类标签，是**验证次数**。这是工程现实主义的应用：你不是在「整理知识」，是在「用经验验证认知」。

## 为什么加入「矛盾字段」？

人的思维往往能同时看到事物的两面。传统笔记只记录「一个结论」，但真正的认知往往是「A 是对的，但 B 也是对的，取决于视角」。

`contradicts` 字段不回避这种张力——它把矛盾显性化，让框架不是因为「没看到反面」而坚固，而是因为**把反面也装进去了**而坚固。

## 为什么做「世界观校准」而不只是「方向锚定」？

方向锚定是「我知道我要去哪」。但真正的问题是——在路上会不断问自己「这真的是我要去的方向吗？」这不是动摇，是校准。

校准回路的设计原则：晴川不做裁判，只做镜子。对比「你说了什么」和「你做了什么」，把偏离呈现出来，让用户自己做判断。偏离可能意味着需要拉回来，也可能意味着方向需要更新——两种都是成长。

## 为什么 MOC 是「认知地图」而不是「目录索引」？

传统的目录索引 = 一个文件夹里有什么文件。但知识库有三层、200+ 篇笔记、跨层生长路径——一个文件列表毫无用处。

MOC 的核心设计原则：**用阅读路径替代文件列表。** 进入一个目录时，MOC 告诉用户的是「如果你想解决 X 问题，读 A→B→C」——不是「这里有 20 个文件」。

MOC 的另一个隐藏功能：**暴露知识缺口。** 如果一条阅读路径写到一半发现缺了某个环节的笔记，MOC 本身就成了待办清单。
